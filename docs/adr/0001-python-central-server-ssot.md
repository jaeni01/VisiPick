# 0001. Python 중앙 서버를 상태의 단일 소유자(SSOT)로 둔다

## 상태
Accepted (2026-05, 프로젝트 설계 단계)

## 맥락
5인 팀이 C# WPF HMI, FastAPI 백엔드, ESP32 게이트·컨베이어, AGV, myCobot 로봇을
각자 개발해 하나의 셀로 통합해야 했다. 공정 상태(검사 결과, 레시피 진행,
트레이 카운트, 비상정지 여부)를 여러 컴포넌트가 각자 들고 있으면
"화면에 보이는 것"과 "실제 설비 상태"가 어긋나는 불일치가 생기고,
통합 단계에서 어느 쪽이 맞는지 판별할 수 없게 된다.

## 결정
Python 중앙 서버(`src/core/state_machine.py`)를 시스템 상태의 유일한
소유자(Single Source of Truth, ISA-95 Level 1 Cell Controller 위치)로 둔다.
- 공정 상태 전이(IDLE/RUNNING/TRAY_TRANSFER/EMERGENCY_STOP 등)는 state_machine 만 수행한다.
- HMI(WPF)는 판단 없이 표시만 한다 — 제어 요청은 명령(MQTT/REST)으로 중앙 서버에 위임한다.
- 검사·판정·게이트·로봇·AGV 명령 분배와 DB 영속화가 모두 이 프로세스에서 출발한다.

## 검토한 대안
- **HMI(WPF)가 제어 주체**: 화면과 제어가 한 곳이라 개발 초기엔 빠르지만, 헤드리스
  운전(HMI 없이 설비만 가동)이 불가능하고 C# 쪽에 장치 프로토콜 5종이 모두 얽힌다.
- **장치별 자율 분산 제어**: 각 장치가 독립 판단하면 단일장애점은 없지만, 게이트
  타이밍·트레이 이재처럼 여러 장치가 맞물리는 시퀀스의 정합성을 보장하기 어렵다.

## 결과
- 화면-실제 불일치가 구조적으로 제거됐고, HMI 없이 `tests/auto_test` 헤드리스
  풀 사이클 검증이 가능해졌다.
- 트레이드오프: 중앙 서버가 단일장애점이 된다. 이 프로세스가 죽으면 셀 전체가
  멈춘다 — 데모 셀 규모에서는 감수하고, 비상정지를 "일시정지"로 설계해(ADR 0005)
  재시작 비용을 낮추는 것으로 보완했다.
