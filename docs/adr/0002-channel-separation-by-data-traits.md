# 0002. 데이터 특성별로 통신 채널을 분리한다 (REST/MQTT/MJPEG/TCP/Serial)

## 상태
Accepted (2026-05, 프로젝트 설계 단계)

## 맥락
셀 하나에 성격이 다른 트래픽이 공존한다: HMI 의 제어 명령·조회, AGV 상태
이벤트, 카메라 실시간 영상, 로봇 모션 설정점, ESP32 게이트·컨베이어 제어.
명령은 요청-응답이 확실해야 하고, 상태는 여러 구독자에게 뿌려져야 하며,
영상은 대용량 연속 스트림이다. 이를 한 채널에 섞으면 대용량 영상이
명령·상태를 밀어내는 등 서로를 막는다.

## 결정
데이터의 요구 특성에 따라 채널을 나눈다.
- **REST/WebSocket (FastAPI :8000)** — HMI 제어 명령·조회: 요청-응답 보장 필요
- **MQTT (:1883)** — 검사 결과·시스템 이벤트·AGV 상태: 1:N 브로드캐스트, 발행-구독
- **MJPEG (HTTP)** — 카메라 영상: 대용량 연속 스트림, 최신 프레임만 의미 있음
- **Ethernet TCP (:9000)** — myCobot: 0.05초 주기 보간 설정점의 저지연 연속 전송
- **USB Serial (115200)** — ESP32 게이트·컨베이어: 배선 단순, 지연 일정

## 검토한 대안
- **전부 MQTT 단일화**: 브로커 하나로 운영은 단순하지만, 요청-응답 시맨틱과
  대용량 영상 스트림에 부적합하고 브로커가 병목·단일장애점이 된다.
- **전부 REST**: AGV 상태처럼 여러 곳이 구독해야 하는 이벤트를 폴링으로
  구현하게 되어 지연과 트래픽이 커진다.

## 결과
- 채널별 요구 특성이 맞아 영상이 명령을 막거나 상태 이벤트가 유실되는 문제가
  없었고, 컴포넌트별 팀원 분담 경계도 채널 경계와 일치해 통합이 수월했다.
- 트레이드오프: 프로토콜 5종의 스펙·포트·장애 모드를 모두 문서화해야 한다
  (docs/MESSAGE_SPEC.md, docs/MQTT_Schema.md). 통신 규약 불일치가 통합 단계
  최대 병목이었다는 경험이 이 문서화 비용의 근거다.
